# 기획 결정 사항 (2026-05-03)

## 작업 내용
백로그 전체 정리 및 핵심 기획 결정 사항 확정

## 핵심 결정 사항

### 인스타그램 파이프라인 (방식 C 채택)
- 인스타 크롤링 = 캡션 일부(OG description) + 이미지 1장(OG image)만 저장
- 장소 연결은 반드시 네이버 지도 검색으로만 처리 (자동 매핑 없음)
- 플로우: 인스타 URL 입력 → 캡션/이미지 저장 → 네이버 지도에서 장소 선택 → Spot 저장 (한 번에 완료 필수, 임시 저장 없음)
- WHY: 서버(Railway)에서 크롤링하므로 사용자 인스타 세션 없음 → location_id 추출 불가
  비로그인 전체 캡션 파싱은 Instagram 봇 감지로 불안정 → OG 메타가 가장 신뢰성 높음

### 공간 DNA
- 4개 축 확정: 붐빔↔여유 / 차분↔화려함 / 최신↔빈티지 / 고급↔가성비
- 분석 주체: AI 팀원이 place_reviews + place_images 데이터 분석
- 연동 방식: AI팀이 DB에 직접 write (place_space_dna 테이블)
- 백엔드 역할: place_space_dna, user_space_dna 조회 API만 구현

### 인증
- 이메일 로그인 + 카카오 OAuth 모두 지원
- 동일 이메일로 두 계정 존재 시 kakao_id 연결로 병합 처리

### 창고 멤버 초대
- 방식: 링크 공유 (토큰 기반 URL)
- 플로우: 수락/거절 API 구현 (즉시 추가 아님)

### 공개 창고
- 탐색: 검색 + 알고리즘 피드
- 저장(스크랩) 기능 구현 예정
- 소셜 기능: 좋아요/팔로우 없음, 스크랩만

### 반경 검색
- 사용자 현재 위치 기반
- 내 저장 장소(Spot이 있는 Place)만 반환 (공개 장소 포함 안 함)

### 리뷰
- 사용자 직접 작성 리뷰 기능 없음
- place_reviews 테이블은 네이버 지도 raw_data에서 수집된 데이터 용도

### 클라이언트
- 모바일 앱 (iOS/Android)
- 프론트엔드 팀원 별도 존재

## 배운 점
- Instagram 크롤링은 서버사이드이므로 클라이언트 세션과 무관 → 처음부터 이 한계를 명확히 설계에 반영해야 함
- AI팀이 DB 직접 접근하는 방식은 백엔드 공수를 줄이지만 스키마 변경 시 사전 협의 필수
- MVP 2주 일정에서 인스타+네이버 파이프라인, DNA 조회, 유저 DNA 업데이트가 핵심
